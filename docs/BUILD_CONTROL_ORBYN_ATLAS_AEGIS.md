## Mission

Build a governed operational intelligence stack where:

- **Orbyn Atlas** observes signals, structures context, models risks and scenarios, and produces executive briefings.
- **Aegis AI Assurance Guardian** evaluates, authorizes, limits, adversarially tests, and documents AI deployments.
- **Orbyn Adaptive Agent Architecture (OAAA)** designs and assembles agents for a company only under predefined rules, permissions, evidence, human approval, and Aegis authorization.

## Non-negotiable doctrine

1. Orbyn Atlas generates operational intelligence.
2. Aegis decides whether the AI use case may proceed.
3. ARIA tests behavior under adversarial pressure.
4. Vigilance enforces technical permissions and logs.
5. DataForge preserves sources, outputs, evidence, versions, and learning.
6. Sovereign Vault protects sensitive doctrine and documents.
7. No agent may self-authorize, elevate its own permissions, or deploy itself to production.
8. No MVP component may connect to or act on critical infrastructure.
9. Every critical recommendation remains decision support for a named human owner.

## Initial product boundary

### Build now

- Kernel minimum data model.
- Atlas Case Register.
- Atlas Signal, Risk, Scenario, Stakeholder, and Briefing outputs.
- Aegis Use Case Register.
- Aegis Clearance Score.
- Decisions: APTO, APTO CON CONTROLES, NO APTO TODAVÍA, REQUIERE REVISIÓN ESPECIALIZADA.
- Guardrails Plan.
- ARIA basic test register.
- Proof Receipt minimum.
- Output Vault.
- OAAA Agent Blueprint proposal flow.
- Human approval gates.

### Do not build yet

- Autonomous action on real infrastructure.
- Self-deploying agents.
- Automatic permission elevation.
- Full SOC, SIEM, SOAR, or enterprise GRC suite.
- Full digital twins with real-time critical feeds.
- Convergence Metaverse.
- Claims of legal certification or automatic regulatory compliance.

## Delivery discipline

Every work item must include:

- defined owner;
- acceptance criteria;
- tests;
- security assumptions;
- human approval point;
- evidence generated;
- rollback path.

## Definition of Done

A module is done only when it:

1. works locally;
2. passes automated tests;
3. validates input and handles failure safely;
4. records version and owner;
5. generates or links evidence;
6. contains no hidden autonomous production action;
7. is documented;
8. has been reviewed before merge.
